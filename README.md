# SOC Detection Requirements

A version-controlled repository serving as the **source of truth** for Blue Team detection requirements. Browse, search, and track detection coverage via GitHub Pages, while keeping everything auditable in Git.

**Key Features:**
- 📋 **Detection Inventory:** Catalog of organized detections mapped to ATT&CK tactics and techniques
- 🎯 **Priority Queue:** AI-powered ranking of detections by implementation priority for internal/air-gapped networks (reverse kill chain order)
- 🗺️ **Live ATT&CK Mapping:** Automatically synced with MITRE enterprise ATT&CK framework
- 🔍 **Full-Text Search:** Find detections by tactic, detection name, or coverage gaps
- 🚀 **Operational Framework:** Every detection tagged with SPEED framework categories for implementation planning

## Live MITRE Mapping

The repository includes a live ATT&CK mapping that correlates:

- ATT&CK tactics
- ATT&CK techniques
- ATT&CK groups
- ATT&CK software
- local detections in this repository

The mapping is sourced from MITRE enterprise ATT&CK STIX:

- `https://raw.githubusercontent.com/mitre-attack/attack-stix-data/master/enterprise-attack/enterprise-attack.json`

Generated artifacts:

- `data/mitre_live_mapping.json`
- `docs/data/mitre_live_mapping.json`

## 🚀 Getting Started

### Adding a New Detection

1. Copy `templates/detection-template.md` into the appropriate category folder under `detections/`.
2. Name the file using the detection ID (e.g., `DET-E-001.md`).
3. Fill in all required frontmatter fields:
   - **Metadata:** `detection_id`, `title`, `category`, `status`, `network_applicability`
   - **ATT&CK mapping:** `mitre_tactics`, `mitre_techniques`, `threat_actors` (all as YAML lists)
   - **Operational Context:** `speed_framework` (list of relevant categories from the SPEED Framework section above)
4. Fill in markdown sections — Description, Use Case, Data Sources, Logic/Query, and References.
5. Ensure `status` is one of: `Planned`, `In Testing`, `Provisioned`, `AI Suggested`, or `Deprecated`.
6. Ensure `network_applicability` includes `both` or `air_gapped` if the detection should appear in the Priority Queue.
7. Open a Pull Request; once merged to `main`, the site rebuilds automatically and the detection appears in:
   - Detections index
   - ATT&CK Live Mapping
   - Priority Queue (if applicable)
   - Search corpus

### Detection ID Convention

| Prefix | Category |
|--------|----------|
| `DET-E-XXX` | Endpoint |
| `DET-N-XXX` | Network |
| `DET-I-XXX` | Identity |
| `DET-C-XXX` | Cloud |

### Status Values

| Status | Meaning |
|--------|---------|
| `Planned` | Requirement identified, not yet built |
| `Provisioned` | Detection is live in the SIEM/EDR |
| `In Testing` | Detection deployed but undergoing validation |
| `AI Suggested` | Generated by AI | Deprecated | No longer maintained |

## Priority Queue Dashboard

The **Priority Queue** dashboard ranks detections by implementation priority for internal and air-gapped networks using a multi-factor scoring system:

- **Status Weight**: `Planned` (400) > `AI Suggested` (300) > `In Testing` (200) > `Provisioned` (100)
- **Network Applicability Weight**: Detections targeting `air-gapped` networks (80 pts) or `both` (60 pts)
- **SPEED Framework Weight**: Prioritizes operational detection families:
  - `host` (40), `internal-network` (35), `access-control` (30), `self-monitoring` (25)
  - Lower weight for broader categories: `policy` (20), `application` (15), etc.
- **Reverse Kill Chain Tactic Weight**: Prioritizes tactics indicating active compromise:
  - `Exfiltration` (TA0040, 50pts) > `Collection` (TA0011, 48pts) > `Lateral Movement` (TA0010, 46pts) > ... > `Reconnaissance` (TA0001, 10pts)
  - This assumes the threat actor has already passed the perimeter and focuses on detecting active intrusion

**Access the dashboard:** Navigate to the **Priority Queue** tab in the documentation site.

## SPEED Framework

All detections include a `speed_framework` frontmatter field mapping to relevant operational priorities:

- `host` — host-based process/behavior monitoring
- `internal-network` — internal network traffic analysis
- `access-control` — user/authentication event detection
- `self-monitoring` — system self-monitoring and anomaly detection
- `policy` — policy violation and compliance events
- `application` — application-layer event detection
- `threat-intelligence` — threat intelligence correlation
- `attack-pattern` — known attack pattern matching
- `cloud` — cloud infrastructure detection
- `compliance` — regulatory compliance monitoring
- `physical` — physical security events
- `mobile` — mobile device events
- `wireless-network` — wireless network events
- `threat-modeling` — threat model-driven detection
